- `Prob151_review2015_fsm`
- `Prob152_lemmings3`

This is the dominant compile-failure class and is tightly concentrated in enum-based state-machine
implementations.

### Invalid procedural assignment target

`Prob039_always_if` assigned `out_always` procedurally while it was declared as a wire. Icarus
reported `out_always is not a valid l-value`. The historical adapter labeled this as
`IVERILOG_UNCLASSIFIED_FAILURE`, which stopped the batch. The diagnostic classifier has since been
fixed so source-bound errors of this form are classified as candidate `COMPILE_ERROR` rather than
infrastructure failure.

### No generated compile unit

`Prob147_circuit10` completed its Agent process without creating a `.sv` or `.v` compile unit and
was rejected with `NO_COMPILE_UNIT`.

### Not executed

`Prob040_count10` has no candidate result from this full run. It was skipped after the historical
Prob039 classification error and should be rerun separately before it is assigned a model outcome.

## Dataset Verification-Interface Error

`Prob099_m2014_q6c` passed the candidate-only compile. The hidden combined compile then failed for
both modules because the testbench instantiated ports `Y2` and `Y4`, while the public specification,
reference module, and candidate expose `Y1` and `Y3`. This case should be classified as invalid
fixture/verification input, not a model functional failure.

## Interpretation and Limitations

- `authoritative: false` means these results are local, non-authoritative benchmark simulations,
  not formal RTL Gate evidence.
- The batches predate automatic mismatch root-cause diagnosis and the current
  `common-guidance.md` prompt injection. No retrospective model calls were made for this report.
- The 21 mismatch cases have counts, affected output hints, and first-mismatch times in their batch
  evidence, but they do not yet have the new structured diagnosis conclusions.
- `Prob040_count10` requires a separate run to obtain a candidate outcome.
- `Prob099_m2014_q6c` requires a dataset/provider disposition before it can participate in a fair
  functional metric.

The evaluator now reports this class separately as `verificationInvalid`, makes the functional
batch `INVALID`, and does not include it in `functionalFailed`. The historical batch JSON predates
that correction; the adjusted counts in this report already apply the corrected interpretation.
